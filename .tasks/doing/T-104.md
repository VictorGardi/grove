## Description

When a user opens a free terminal tab (Cmd+J) in workspace 1, switches to workspace 2, and switches back, the terminal tab is visually present but unresponsive. Interactive programs (e.g., Copilot CLI) inside the terminal appear dead — they don't respond to input and produce no visible output.

**Root cause:** `TerminalTabView` calls `term.dispose()` on unmount (triggered by the workspace-scoped tab filter in `TerminalPanel`). This destroys the xterm instance while the underlying PTY process stays alive. On remount, a fresh xterm is created with an empty buffer. PTY output continues flowing to the main process but is silently dropped because the xterm is no longer in `xtermRefs`. Keystrokes still reach the PTY but the user sees nothing.

**Fix:** Stop calling `term.dispose()` on unmount. Only call `unregisterXterm(id)` so the xterm stays alive in `xtermRefs` across workspace switches. On remount, detect the existing xterm and reuse it instead of creating a new one. PTY output accumulates in the xterm buffer while the tab is unmounted and becomes visible on return. Keystrokes work immediately because `window.api.pty.write` doesn't need the mount callback.

## Definition of Done

- [x] Terminal scrollback (output history) is preserved after switching away from and back to a workspace with a free terminal tab open
- [x] A free terminal tab remains interactive immediately after returning to its workspace (user can type and see output without pressing a key to restart)
- [x] If the PTY process exits while the tab is unmounted, the "Process exited (code N)" message is visible when the tab remounts
- [x] When a free terminal tab remounts, it reuses the existing xterm instance (no new empty xterm is created)
- [x] Closing a free terminal tab properly cleans up its xterm instance (no orphaned xterm in `xtermRefs`)
- [x] Resize (fit) continues to work correctly when a tab remounts after being unmounted

## Notes for implementation

- The `onData` callback for user keystrokes lives in TerminalTabView's mount effect and is not re-registered on remount. This is fine because `window.api.pty.write(id, data)` sends keystrokes directly to the main process without needing the callback. PTY output (routed through `initTerminalListeners` → `xtermRefs`) is preserved because the xterm stays in `xtermRefs`.
- Pre-existing: removing a workspace does not clean up its terminal xterms from `xtermRefs`. This is a separate leak not introduced by this fix — track separately if needed.
- The deadSet Zustand state may be stale on remount (doesn't reflect a PTY that died while unmounted). The "Process exited" message is written to the xterm buffer by the `onExit` handler at the time of death, so it will display on remount regardless of deadSet state.
