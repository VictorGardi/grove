## Description

There are two generations of agent session architecture in Grove:

**Generation 1 (legacy — to be removed):**
`plan:send` IPC → `AgentRunner` → `TmuxSupervisor` → headless tmux session named `grove-exec-*` or `grove-plan-*`. Sets `execTmuxSession` / `planTmuxSession` in task frontmatter. Output was streamed via `plan:chunks` → `usePlanStore`. The rendered message array for `execute:${taskId}` was never shown in any UI component. `planTmuxSession` has zero call sites that could ever set it — it is dead from creation.

**Generation 2 (current — to keep):**
`taskterm:create` IPC → interactive opencode TUI in an xterm.js terminal tab. Sets `terminalPlanSession` / `terminalExecSession` in frontmatter. This is the only agent output path visible in the UI.

**The bug:**
When a task is dragged to "doing", `handleDragToDoing` (Board.tsx ~line 450) calls `window.api.plan.send(...)`, which starts a Gen-1 headless agent and writes `execTmuxSession` to frontmatter. When the user then opens the task detail panel, `TaskDetailPanel.tsx:654` reads `task.terminalExecSession` — a different field that Gen-1 never sets. `TaskTerminal`'s mount effect sees `sessionMode === "exec"` with no session and auto-starts a second Gen-2 agent via `taskterm:create`. Two simultaneous OpenCode agents now run against the same task.

**The fix:** Replace the Gen-1 `plan.send` call in `handleDragToDoing` with `taskterm:create`, so the drag-to-doing flow starts a Gen-2 interactive terminal session and writes `terminalExecSession`. Remove all remaining Gen-1 code and frontmatter fields.

### 1. `Board.tsx` — `handleDragEnd` (line 164–167)

The `isRunning` guard currently checks `usePlanStore.sessions[execSessionKey].isRunning`. After removing `initSession` from the new flow, this store entry is never populated, making the guard permanently `false`. Replace with: `if (task.terminalExecSession) return;` — a Gen-2 session already in frontmatter means one is running.

### 2. `Board.tsx` — `handleDragToDoing` cleanup block (lines 317–339)

Remove the Gen-1 cleanup: `plan.cancel` call, `usePlanStore.clearSession`, and the `execSessionId`/`execTmuxSession` null-write. Replace with a `taskterm.kill` + `updateTask({ terminalExecSession: null })` guard for re-drag, mirroring the existing `terminalPlanSession` kill block at lines 341–351.

### 3. `Board.tsx` — `handleDragToDoing` Gen-1 auto-start block (lines 394–477)

Remove: `usePlanStore.initSession(execSessionKey, ...)`, `window.api.plan.send(...)`, the success/failure branches that update `usePlanStore`. Replace with a `window.api.taskterm.create(...)` call passing `worktreeAbsPath` as `cwd`, the execution prompt as `initialContext`, and the task's exec agent/model config.

### 4. `Board.tsx` — `handleDragToDone` (lines 513–523)

Already kills `terminalExecSession` on done — no change needed.

### 5. `TaskDetailPanel.tsx` — `DebugPanel` (lines 35–88)

### 6. `TaskDetailPanel.tsx` — line 654

Set `initialSessionName = task.terminalExecSession ?? null`. The Gen-1 fallback (`?? task.execTmuxSession`) is no longer needed once Gen-1 is removed.

### 7. `TaskCard.tsx` — status badge conditions (lines 80–141)

- `isPlanErrored` (line 135): same issue with `task.planTmuxSession != null`. Rewrite to `task.terminalPlanSession != null`.
- `isPlanWaitingWithTmuxAlive` (line 107): remove `task.planTmuxSession` half.

### 8. `useWorkspaceStatus.ts` — polling loop (lines 135–140, 237–242)

Remove the `else` branch that calls `window.api.plan.tmuxCheck(...)`. After removing Gen-1, all remaining sessions start with `grove-term-*`, so the `isTerminalSession` check and the `else` branch are dead.

### 10. `App.tsx` — startup session init (lines 129–167)

### 11. `main/ipc/plan.ts`

Remove the `plan:send` exec-mode branch. No renderer call sites for plan-mode `plan.send` exist either — gut the entire `plan:send` handler. Also remove `plan:reconnect` (zero renderer call sites). Remove `plan:cancel` exec-mode branch (or entire handler if plan-cancel is also gone). Remove `plan:saveSession` if it is only called by the Gen-1 exec output path.

Decide whether `execSessionId` and `planSessionId` are also removed (they are Gen-1 artifacts written by `plan:saveSession` and read back in `App.tsx:initSession`) — if yes, remove; if kept for legacy frontmatter compatibility, mark as `@deprecated`.

Remove parsing/serialization of `execTmuxSession` and `planTmuxSession` (and `execSessionId`/`planSessionId` if removed from the type).
