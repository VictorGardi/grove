## Description

Audit and optimize renderer-side performance to eliminate input lag during chat interaction and reduce background IPC/render churn caused by polling, streaming, and unnecessary re-renders.

### Root causes identified

**Chat input lag (highest impact)**

- `PlanChat.tsx` `handleInputChange`: every keystroke triggers a synchronous `scrollHeight` layout reflow, 3 `useMemo` re-evaluations (regex), and a `getBoundingClientRect` read for dropdown positioning — all with no debouncing.
- `usePlanStore.applyChunk`: every streaming token creates new object references for the entire `sessions` tree and calls Zustand `set()`, producing high-frequency renders that compete with keyboard events on the JS thread.
- `PlanChat.tsx` has a single coarse Zustand selector (`sessions[sessionKey]`) that re-renders the entire component on every streaming chunk. Fine-grained selectors are missing.
- `ChatMessage` has no `React.memo`, so all historical messages re-render on every streaming chunk.
- `applyChunk` always spreads the entire `messages` array into a new array, giving every message object a new reference identity — making `React.memo` on `ChatMessage` ineffective without also fixing reference stability.
- No list virtualization: all messages are rendered to the DOM with a plain `.map()`.

**Background noise (compounds the above)**

- `TaskCard.tsx`: two `setInterval` timers per card (5 s each) each spawn a `tmux has-session` subprocess. 10 doing-tasks = 20 subprocess spawns every 5 seconds, running continuously.
- `useWorkspaceStatus.ts`: self-reinforcing 2 s poll loop — cache update (`useState`) → new `useCallback` reference → `useEffect` re-runs → interval reset → repeat.
- `useTerminalStore` `setIdle`: Zustand `set()` called on every terminal output byte with no rate limiting.
- `App.tsx`: workspace-switch `useEffect` is registered twice (exact duplicate), doubling IPC calls and store updates on every workspace switch.
- `pty.ts` line 121: `console.log(JSON.stringify(data))` fires on every keystroke in any terminal panel (synchronous, on the main-process event loop).

**Also notable**

- `TaskDetailPanel.tsx` `renderMarkdownPreview`: 15 regex replacements over the full document string run on every render with no `useMemo`.
- `tmuxSupervisor.ts`: 20 Hz file polling per active agent session.
- `tmuxAliveCache` in `useWorkspaceStatus`: once a session is cached as alive it is never re-checked (no TTL/invalidation), so a dead session can remain marked alive indefinitely.

### Proposed solutions

1. **Debounce `handleInputChange` keyboard path** (`PlanChat.tsx`): Keep `setInputText` immediate (controlled input). Extract the side-effect logic (regex, `setShowDropdown`, `fetchTree`) into a separate function and debounce it at ~100ms. Move `resizeTextarea` to a `requestAnimationFrame` callback so the `scrollHeight` read happens after paint. Ensure debounce only applies to the _open_ logic for dropdowns — the _close_ path (e.g. clearing dropdown on backspace past trigger) must remain immediate to avoid a window where Enter fires while a stale dropdown is still visible.

2. **Fix dropdown position layout thrashing** (`PlanChat.tsx`): Wrap the `getBoundingClientRect` read in the dropdown-position `useEffect` inside a `requestAnimationFrame` callback. Document that this introduces a one-frame lag on first appearance, and verify the dropdown still positions correctly on initial open.

3. **Add fine-grained Zustand selectors in `PlanChat`** (`PlanChat.tsx`): Replace the single `usePlanStore(s => s.sessions[sessionKey])` selector with multiple narrow selectors (e.g. `messages`, `isRunning`, `sessionStatus`, `totalTokens`) using `useShallow` for arrays, so unrelated streaming updates do not re-render the whole component.

4. **Fix `applyChunk` reference stability + memoize `ChatMessage`** (`usePlanStore.ts` + `PlanChat.tsx`): Refactor `applyChunk` to reuse existing message object references for messages it does not modify (only rebuild the object for the message actively being updated). Then wrap `ChatMessage` with `React.memo`. Without reference stability in the messages array, `React.memo` has no effect on historical messages.

5. **Throttle `applyChunk` streaming updates to ~60 fps** (`usePlanStore.ts`): Buffer incoming _content_ chunks (`text`, `thinking`, `tool_use`) in a module-level or ref-based queue and flush to Zustand state via a single `set()` call scheduled with `requestAnimationFrame` — at most once per frame. Control-flow chunks (`done`, `session_id`, `replay_done`, `user_message`) must remain synchronous and are not buffered. Buffered chunks must be applied in arrival order within each frame flush.

6. **Memoize `renderMarkdownPreview`** (`TaskDetailPanel.tsx`): Wrap the call in `useMemo` with `[rawContent]` as the dependency so the 15-regex chain only runs when content actually changes.

7. **Remove duplicate `useEffect` in `App.tsx`**: Delete the duplicate workspace-switch effect (lines 113–121). For the `initSession` effect, audit whether it needs to re-run when session metadata (e.g. `planTmuxSession`, `planSessionId`) changes — not just when task IDs change — before changing its dependency array, to avoid suppressing legitimate re-runs.

8. **Consolidate tmux polling** (`TaskCard.tsx` + `useWorkspaceStatus.ts`): Remove the two per-card `setInterval` timers from `TaskCard.tsx`. Promote the tmux alive cache to a shared Zustand slice (or extend `useWorkspaceStatus`) that stores _per-task, per-mode_ liveness (`{ taskId, mode: 'exec'|'plan', alive: boolean }`). `TaskCard` reads from this shared store reactively. Ensure the replacement poller covers both exec and plan tmux sessions for all doing-tasks.

9. **Fix `useWorkspaceStatus` self-reinforcing poll loop + add cache TTL** (`useWorkspaceStatus.ts`): Move `tmuxAliveCache` from `useState` to `useRef` so cache updates don't recreate the `checkTmuxSessions` callback. Introduce a separate lightweight render-trigger (e.g. an incrementing counter in state) that is updated after cache writes so derived status stays current. Additionally, add a TTL or explicit invalidation to the cache so sessions that die are eventually re-detected.

10. **Debounce `setIdle` in `useTerminalStore`**: Apply a trailing-edge debounce (~100ms) to the `setIdle(id, false)` call inside the `pty:data` listener so Zustand `set()` is not called on every output byte. Note: fast commands that complete in <100ms may not show the "running" indicator — this is acceptable; the DoD should reflect this.

11. **Remove `console.log` from `pty.ts` write hot path**: Delete or guard behind a `DEBUG` env flag the `console.log("[PtyManager.write]...", JSON.stringify(data))` at `pty.ts:121`.

12. **(Stretch) Virtualize the message list** (`PlanChat.tsx`): Use `@tanstack/react-virtual` (preferred for variable-height support) or `react-window` `VariableSizeList` to render only visible `ChatMessage` nodes. Heights must be measured or estimated; document that scroll-position correction after height measurement is expected behavior.

### Manual verification checklist (not a DoD gate)

- Typing rapidly in the chat input feels responsive with no perceptible lag.
- Streaming agent output does not block or delay keyboard input.
- Board with 10+ doing-tasks shows no visible jank during normal interaction.
- Terminal tab bar idle/running indicator behaves correctly for commands taking >100ms.
- A session that stops running is eventually detected as not alive (cache invalidation working).
