## Description

A minimal Grove CLI that reuses the shared runtime layer from the Electron app. The CLI is thin and focused — it does not duplicate UI logic or create a second workflow system. It exposes core runtime actions for power users and automation.

### Commands

**1. `grove run [agent] [message]`**

- Starts a task runtime and launches an interactive agent session
- Agents: `opencode` (default), `copilot`
- Options:
  - `--branch <name>` — create or use a git worktree for the task
  - `--task <id>` — specify existing task ID to run (e.g., `T-004`)
  - `--model <name>` — specify model to use (optional)
- If `--branch` is not provided, runs against the root repo (default behavior)
- If task does not exist, may create a new task record or attach to a new one
- **Uses containerization when available** (see T-175: add containerization)

Examples:

- `grove run opencode "fix the login bug"`
- `grove run --branch my-test-branch opencode "implement feature"`
- `grove run --task T-004 opencode`

**2. `grove attach <task-id>`**

- Attaches to an existing running task/session
- Restores interactive terminal access to the agent runtime
- **Always uses tmux** — spawns new tmux session attached to the task
- Works for both container sessions and host sessions
- **Handles containerized sessions** — attaches to tmux session running inside container

Example: `grove attach T-004`

**Task ID Resolution**

- Task IDs are scoped to a workspace
- Workspace is resolved first: `--workspace` flag > auto-detect `.tasks/` in CWD > global config
- Task is then looked up within that resolved workspace
- If the same task ID exists in multiple known workspaces, CLI errors asking user to specify `--workspace` to disambiguate

Examples:

- `grove attach T-004` — resolves workspace first, then looks up T-004 in that workspace
- `grove attach --workspace /path/to/repo T-004` — explicitly uses that workspace

**3. `grove list`**

- Lists tasks with key metadata:
  - Task ID
  - Title
  - Workspace/repo path
  - Status (backlog/doing/review/done)
  - Runtime state (local/container)
  - Whether a session is active
  - Container status (if applicable)
- Human-readable terminal output

### Containerization Integration

The CLI leverages containerization from T-175 (add containerization):

- When containerization is enabled in workspace config:
  - Tasks run inside Docker/Podman containers
  - Uses `.devcontainer/devcontainer.json` if present
  - Falls back to default Ubuntu image if no devcontainer config
  - Mounts repo/worktree path into container at `/workspace`
- When containerization is disabled or unavailable:
  - Falls back to local execution (host mode)
- Container lifecycle managed by `ContainerService` from runtime layer

### Workspace Discovery

Priority order (CLI must work even if config doesn't exist):

1. `--workspace <path>` — explicit flag (highest priority)
2. Auto-detect `.tasks/` directory in current working directory (primary)
3. Shared global config at `~/.grove/config.json` (fallback)

When CLI finds a `.tasks/` folder, it should auto-add the workspace to the global config so Electron app and CLI share the same known workspaces.

**Task ID Resolution**:

- Task IDs are scoped to a workspace — CLI always resolves workspace first, then looks up task within that workspace
- If multiple known workspaces contain the same task ID, CLI errors asking user to specify `--workspace` to disambiguate

### Implementation Notes

- Reuse runtime services from `src/runtime/` (taskService, agentService, gitService, tmuxService, containerService)
- CLI-specific config handling: use `~/.grove/config.json` instead of Electron's `app.getPath("userData")`
- Runtime services are pure Node.js (no Electron imports) — safe to reuse directly
- **Always use tmux** for interactive sessions — no fallback to direct spawn (tmux is required)
- Exit codes: 0 = success, 1 = general error, 2 = workspace not found, 3 = task not found in workspace
- Verify agent binary exists (`opencode`, `copilot`) before attempting to run
- Task ID lookup is always scoped to resolved workspace (not global)

### Design Principles

- The CLI is NOT a separate workflow system — it operates on the same task model as the UI
- Optional worktree usage (not forced) — root repo mode is default
- Concise, useful output — suitable for both human and automation use
- Works even when the Electron app is closed
- Containerization is optional and configurable per workspace
