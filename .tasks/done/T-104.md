## Description

When a user opens a free terminal tab (Cmd+J) in workspace 1, switches to workspace 2, and switches back, the terminal tab shows a blank fresh session — all previous output is gone. The user must also click the terminal tab to make it active before they can interact with it.

**Root cause (Part 1 — blank session):** `TerminalTabView`'s unmount cleanup correctly does _not_ call `dispose()` — the xterm instance survives in `xtermRefs` across the workspace switch. However, the _remount_ reuse logic in the mount effect tries to locate the xterm's DOM element using `existingTerm._core._terminalEl`, a private internal property that does not exist in `@xterm/xterm` v6 (confirmed: 0 occurrences in the compiled bundle). Because `_terminalEl` is always `undefined`, the guard `if (core && termElement)` always fails and falls into the else branch, which calls `existingTerm.dispose()` and `unregisterXterm(id)`. A fresh xterm with an empty buffer is then created, discarding all previous output.

**Root cause (Part 2 — must click tab):** `activeTabId` in `useTerminalStore` is a single global value shared across all workspaces. After switching to workspace 2, `activeTabId` may point to workspace 2's terminal tab. On returning to workspace 1, workspace 1's terminal renders with `visible=false` (display:none) and requires a click to activate. The workspace state save/restore (`workspaceTerminalStates` in `useWorkspaceStore`) persists `terminalPanelOpen` per workspace but not `activeTabId`.

**Fix (Part 1):** In `TerminalTab.tsx`, replace `existingTerm._core._terminalEl` with `existingTerm.element` — the public xterm v6 API (`HTMLElement | undefined`) returning the xterm-created `<div class="terminal xterm">` that lives inside the original container. On remount, detach this element from its old (now-detached) container and append it to the new `containerRef.current`. Before loading a new `FitAddon`, dispose the previous one to avoid accumulating multiple addon instances on the same terminal across workspace switches.

**Fix (Part 2):** Extend `TerminalState` in `workspaceTerminalStates` (`useWorkspaceStore`) with `activeTabId: string | null`. Save it in `setActiveWorkspace` alongside `terminalPanelOpen`. Restore it in `restoreTerminalState` by calling `useTerminalStore.getState().setActiveTab(saved.activeTabId)` if the saved tab still exists in `tabs`.

## Definition of Done

- [x] Terminal scrollback (output history) is preserved after switching away from and back to a workspace with a free terminal tab open
- [x] On returning to a workspace, the previously active terminal tab is automatically selected (no click required)
- [x] A free terminal tab is immediately interactive on return — the user can type and see output without needing to press any key to trigger a PTY restart
- [x] If the PTY process exits while the tab is unmounted, the "Process exited (code N)" message written to the xterm buffer at exit time is visible when the tab remounts
- [x] When a free terminal tab remounts, the existing xterm instance is reused — no new `Terminal` object is constructed for the remounting tab
- [x] Closing a free terminal tab disposes its xterm and removes it from `xtermRefs` (regression guard — already correct, must not regress)
- [x] Resize (fit) continues to work correctly when a tab remounts, and no more than one `FitAddon` is active on a reused xterm at any time (the previous `FitAddon` is disposed before a new one is loaded)

## Notes for implementation

- **xterm element API:** Use `existingTerm.element` (public API, type `HTMLElement | undefined`) to get the xterm-created DOM element on remount. Do NOT use `_core._terminalEl` — this private property does not exist in `@xterm/xterm` v6. `element` is `undefined` before `open()` has been called; for any xterm already in `xtermRefs`, it will always be defined.
- **FitAddon disposal:** On remount in the reuse path, the previous `FitAddon` must be disposed before a new one is loaded. Track the previous addon in a way that survives unmount (e.g., store it in a ref that is not cleared on unmount, or store it alongside the xterm in the registry).
- **activeTabId save/restore:** Add `activeTabId: string | null` to the `TerminalState` interface inside `workspaceTerminalStates` in `useWorkspaceStore.ts`. Save in `setActiveWorkspace` alongside `terminalPanelOpen`. In `restoreTerminalState`, call `useTerminalStore.getState().setActiveTab(saved.activeTabId)` only if the saved id still exists in `tabs` (guard against stale ids from a closed tab).
- **onData re-registration:** The `onData` listener is disposed in the unmount cleanup and re-registered on remount for the reused xterm. This is correct — there will always be exactly one active `onData` listener per mounted `TerminalTabView`.
- Pre-existing: removing a workspace does not clean up its terminal xterms from `xtermRefs`. This is a separate leak not introduced by this fix — track separately if needed.
- The `deadSet` Zustand state may be stale on remount (doesn't reflect a PTY that died while unmounted). The "Process exited" message is written to the xterm buffer by the `onExit` handler at the time of death, so it will display on remount regardless of `deadSet` state.
