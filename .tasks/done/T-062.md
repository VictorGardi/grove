## Description

The "agent running" indicator on board task cards is unreliable. It flickers between "agent running" and "Waiting for input" and shows "Waiting for input" prematurely — even while the agent is actively streaming and the chat view shows "Session active". The indicators need to be a solid source of truth.

### How the state machine works (current)

The board card derives its indicator purely from `usePlanStore`:

- **"agent running"** → `session.isRunning === true`
- **"Waiting for input"** → `!isRunning && messages.length > 0 && (lastExitCode === null || lastExitCode === 0)`
- **"session failed"** → `!isRunning && messages.length > 0 && lastExitCode !== null && lastExitCode !== 0`

`isRunning` is set `true` by `startAgentMessage()` and re-asserted by every `text`/`thinking` chunk. It is set `false` by a `done` or `error` chunk — or by a mount-only `useEffect` in `PlanChat` that fires every time the task detail panel is opened.

`hasSession` (which drives "Session active" in chat) is `!!session.sessionId` — completely independent of `isRunning`. This is why "Session active" and "agent running" can be out of sync: when `isRunning` is wrongly reset to `false`, the chat still shows "Session active" (sessionId unchanged), but the board card shows "Waiting for input".

### Root causes found

#### Bug 1 — CRITICAL: Mount-only `isRunning` reset clobbers live sessions on panel open/close

**File:** `src/renderer/src/components/TaskDetail/PlanChat.tsx:280–286`

Every time the user closes and reopens the task detail panel, `PlanChat` unmounts and remounts (the panel unmounts completely when `clearSelectedTask()` is called). On remount, a `useEffect([], [])` unconditionally resets `isRunning: false` if it's `true`.

The comment says "next chunk from a still-running agent sets `isRunning` back to `true`". That is true — but there is a gap between the reset and the next chunk arriving. During this gap:

- `isRunning: false`
- `messages.length > 0`
- `lastExitCode === null`

→ The board card shows **"Waiting for input"** despite the agent running. This is exactly the observed flicker.

#### Bug 2 — CRITICAL: Reconnect replay never sets `isRunning: true`, causing "Waiting for input" during replay

**File:** `src/renderer/src/components/TaskDetail/PlanChat.tsx:308–376`, `src/renderer/src/App.tsx:199–210`

When the user reopens a task with a live or previously-running tmux session:

1. The mount-only reset (Bug 1) fires first → `isRunning: false`
2. The reconnect effect fires asynchronously and calls `setSessionStatus(sessionKey, "running")` — but does NOT call `setRunning(sessionKey, true)`
3. Chunks replay. Only the FIRST `text` or `thinking` chunk lazily triggers `startAgentMessage()` in `App.tsx`, which sets `isRunning: true`

During the entire period before the first content chunk arrives (IPC round-trip + any `session_id`/`step_start` preamble events), `isRunning` is `false` → "Waiting for input" on the board card, even for a session actively replaying a live run.

#### Bug 3 — CRITICAL: Cancel in tmux path never emits `done` chunk → `isRunning` stuck `true` forever

**File:** `src/main/planManager.ts:360–371`, `src/main/tmuxSupervisor.ts:488–508`

`TmuxPlanRunner.cancel()` calls `this.tmux.kill(tmuxSession)`. `TmuxSupervisor.kill()` calls `stopTailer()` first (which closes the file descriptor and poll timer), then kills the tmux session. Because the tailer is stopped before the `grove_exit` sentinel in the log can be read, no `done` chunk is ever emitted.

Result: `isRunning` stays `true` indefinitely. The card shows "agent running" even after cancel. The only recovery is the next mount-only reset (Bug 1).

#### Bug 4 — MODERATE: Cancel in spawn path emits `done` with `null` exit code → shows "Waiting for input" after cancel

**File:** `src/main/planManager.ts:287–300`

`SpawnPlanRunner.cancel()` calls `run.rl?.close()` which triggers the `rl.on('close')` handler, which emits a `done` chunk with `String(proc.exitCode ?? 0)`. When the process was sent SIGTERM but hasn't exited yet, `proc.exitCode` is `null` → `String(null)` = `"null"` → `parseInt("null", 10)` = `NaN` → `lastExitCode` set to `null`.

With `isRunning: false` and `lastExitCode: null`, `isExecuteWaiting`/`isPlanWaiting` become `true` → the card shows **"Waiting for input"** after cancelling in the spawn path.

#### Bug 5 — MODERATE: `error` chunk does not set `lastExitCode` → shows "Waiting for input" instead of "session failed"

**File:** `src/renderer/src/stores/usePlanStore.ts:224–254`

When an `error` chunk arrives, `isRunning` is correctly set to `false` (line 251). But `lastExitCode` is never set — stays `null`. Because `isExecuteErrored`/`isPlanErrored` require `lastExitCode !== null && !== 0`, they return `false`. The card falls through to `isExecuteWaiting`/`isPlanWaiting` (lastExitCode is null → `true`) → "Waiting for input" instead of "session failed".

### Proposed fixes

#### Fix A — Make mount-only `isRunning` reset conditional on no stored tmux session

**File:** `src/renderer/src/components/TaskDetail/PlanChat.tsx`

Change the mount-only effect (lines 280–286) to only reset when there is definitely no live agent — i.e. when there is no stored tmux session name in frontmatter. If `storedSession` is non-null, the reconnect effect will determine the correct state.

```ts
// Current (always resets):
useEffect(() => {
  if (session?.isRunning) setRunning(sessionKey, false);
}, []);

// Fixed (only resets when no live session is possible):
useEffect(() => {
  const storedSession =
    mode === "execute" ? task.execTmuxSession : task.planTmuxSession;
  if (!storedSession) {
    const s = usePlanStore.getState().sessions[sessionKey];
    if (s?.isRunning) setRunning(sessionKey, false);
  }
  // If storedSession exists: reconnect effect handles isRunning.
}, []);
```

Inside the reconnect effect, when `reconnectResult.data.reconnected === false`, also call `setRunning(sessionKey, false)` to recover any stale `true` value from a previous crashed run.

#### Fix B — Set `isRunning: true` immediately at reconnect start

**File:** `src/renderer/src/components/TaskDetail/PlanChat.tsx`

When `reconnectResult.data.reconnected === true`, immediately call `setRunning(sessionKey, true)` before chunks start arriving. The board card shows "agent running" throughout replay. The `done` chunk from the log sentinel will set `isRunning: false` when replay finishes.

#### Fix C — Emit synthetic `done` chunk on cancel in tmux path

**File:** `src/main/planManager.ts` — `TmuxPlanRunner.cancel()` (around line 360)

`TmuxPlanRunner` needs access to the chunk callback. Store `onChunkCb` on the instance (it is already passed per-run via `RunOpts`). In `cancel()`, emit a `done` chunk with exit code `1` before killing:

```ts
cancel(runKey: string, workspacePath?: string): void {
  if (!this.activeKeys.has(runKey)) return;
  const [mode, taskId] = runKey.split(":");
  // Immediately clear isRunning in the renderer before the process dies
  this.onChunkCb?.(taskId, mode as PlanMode, { type: "done", content: "1" });
  const tmuxSession = buildTmuxSessionName(workspacePath ?? "", taskId, mode as PlanMode);
  this.tmux.kill(tmuxSession);
  this.activeKeys.delete(runKey);
}
```

#### Fix D — Fix spawn-path cancel to emit `done` with exit code `1`

**File:** `src/main/planManager.ts` — `SpawnPlanRunner.cancel()` (around line 287)

Add a `cancelled` flag to the run entry. In `cancel()`, set the flag and emit a synthetic `done` chunk with exit code `1`. In the `rl.on('close')` handler, skip emitting the `done` chunk if `cancelled` is set, to avoid a duplicate:

```ts
// In the run entry, add: cancelled: boolean = false

cancel(runKey: string): void {
  const run = this.activeRuns.get(runKey);
  if (run) {
    run.cancelled = true;
    // Emit synthetic done with definitive exit code before killing
    run.opts.onChunk(run.taskId, run.mode, { type: "done", content: "1" });
    run.rl?.close();
    run.proc.kill("SIGTERM");
    ...
  }
}

// In rl.on('close'):
rl.on("close", () => {
  if (run.cancelled) return; // already emitted done above
  ...emit done as before...
});
```

#### Fix E — Set `lastExitCode: 1` on `error` chunk

**File:** `src/renderer/src/stores/usePlanStore.ts`

In the `error` branch of `applyChunk` (around line 224), return a complete state update with `isRunning: false` and `lastExitCode: 1`, and do not fall through to the generic fallback return:

```ts
} else if (chunk.type === "error") {
  messages[messages.length - 1] = { ...last, text: ..., isStreaming: false };
  return {
    sessions: {
      ...s.sessions,
      [sessionKey]: { ...session, messages, isRunning: false, lastExitCode: 1 },
    },
  };
  // explicit return — does not fall through to the isRunning fallback
}
```

## Definition of Done

- [ ] Closing and reopening the task detail panel while an agent is actively running never causes the board card to switch from "agent running" to "Waiting for input" at any point during the transition
- [ ] The reconnect/replay period (between panel open and first content chunk) shows "agent running" on the board card, not "Waiting for input"
- [ ] Cancelling a running tmux-backed agent transitions the board card away from "agent running" within one render cycle — no stuck indicator, no flicker
- [ ] Cancelling a running spawn-backed agent has the same immediate transition behaviour
- [ ] After an `error` chunk, the board card shows "session failed" not "Waiting for input"
- [ ] The stale-`isRunning` recovery path still works: if the app is restarted after a crash while an agent was running (no tmux session survives, `storedSession` is null), the board card correctly clears "agent running" on next open

- [ ] Manual test: cancel a running agent via the Cancel button; the board card clears "agent running" immediately
- [ ] Manual test: let an agent finish normally; board card transitions from "agent running" → "Waiting for input" (or "ship it") exactly once with no flicker
- [ ] Manual test: let an agent fail with an error; board card shows "session failed" not "Waiting for input"
- [ ] Unit tests for `usePlanStore` state transitions: `done` chunk clears `isRunning` and sets `lastExitCode`; `error` chunk clears `isRunning` and sets `lastExitCode: 1`; `startAgentMessage` sets `isRunning: true`

## Context for agent

The entire signal chain is: tmux log file → `TmuxSupervisor.drain()` → `onChunk` callback → IPC `plan:chunks` → `App.tsx onChunk` → `usePlanStore.applyChunk()` → Zustand reactive selectors → `TaskCard` computed booleans.

The problem is that `isRunning` in the store is manipulated from two directions: the chunk stream (authoritative) and the `PlanChat` mount effect (a "stale state recovery" heuristic that fires too eagerly). There is also no mechanism to set `isRunning: true` at reconnect time before chunks arrive, leaving a window where the store state is incorrect.

- `src/renderer/src/stores/usePlanStore.ts` — state machine; `applyChunk`, `startAgentMessage`
- `src/renderer/src/components/Board/TaskCard.tsx` — derived indicator logic; `isAgentRunning`, `isPlanningRunning`, `isExecuteWaiting`, `isPlanWaiting`
