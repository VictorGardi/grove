## Description

When a task is dragged from the backlog column to the doing column, Grove opens an agent terminal session (opencode or copilot) via `Board.tsx:handleDragToDoing`. The session is created immediately, but prompt injection never happens — the agent starts with no context.

The root cause: `Board.tsx` calls `window.api.taskterm.create(...)` to start the session but then relies on `TaskTerminal` to call `sendInitialContext` later. However, the task panel is intentionally not auto-opened on drag, so `TaskTerminal` never mounts at that point. When the user eventually opens the panel, `TaskTerminal` sees an existing `terminalExecSession` in frontmatter and takes the `reconnectSession` branch — which only reattaches the PTY and never calls `sendInitialContext`.

The fix must make injection happen immediately after the session is created during the drag, without requiring the user to open the task panel. `Board.tsx:handleDragToDoing` should perform the injection directly (polling for agent idle, writing context, sending the prompt), mirroring what `sendInitialContext` does in `TaskTerminal`.

The bug does not affect the manual send flow (text input → Send), because there `TaskTerminal` creates the session itself and calls `sendInitialContext` in the same step.

## Definition of Done

- [ ] When a task is dragged from backlog to doing, the agent receives the initial execution context (built by `buildFirstExecutionMessage`) immediately after the session is ready — without the user needing to open the task panel
- [ ] Context injection does not occur more than once per exec session — if the user opens the task panel after dragging, injection does not fire again
- [ ] The existing manual send flow (text input → Send) continues to work correctly with context injection
- [ ] A persistent mechanism (e.g. a frontmatter field `terminalExecContextSent: true`) tracks whether injection has been completed for the current session, and is cleared when a new exec session is created (e.g. task moved back to backlog then re-dragged to doing)
- [ ] If the agent TUI is not yet ready when injection is attempted, the system waits (idle-polls) and injection still succeeds once the agent is ready
- [ ] If the agent session never signals idle (agent hangs or crashes at startup), injection times out gracefully — an error is logged and the user is not left in a silent broken state
- [ ] App restart is explicitly out of scope: no guarantee of re-injection if the app restarts mid-session
