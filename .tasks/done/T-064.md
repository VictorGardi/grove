## Description

Agent responses in the PlanChat UI appear sparse compared to using opencode directly. When running opencode in its TUI, users see rich output including thinking blocks, tool use events (bash commands, file reads/writes, grep/glob searches, sub-agent tasks), code blocks, and progress indicators. In Grove's PlanChat, only `text` and `thinking` events are rendered — all `tool_use` events from opencode's `--format json` output are silently ignored.

### Root Cause

`parseOpencodeLine()` in `src/main/planManager.ts` only extracts `text`, `thinking`, `session_id`, and `tokens` events. The `tool_use` event type — which contains tool name, input, output, title, exit codes, and timing — is never parsed or forwarded to the renderer.

### Proposed Solution

**Phase 1 — Data Pipeline (backend → store)**

1. Add `tool_use` to the `PlanChunk` union type in `src/shared/types.ts`:

   ```
   { type: "tool_use"; content: string; data: ToolUseData }
   ```

   where `ToolUseData` includes: `tool` (string), `input` (object), `output` (string, truncated to 5KB), `title` (string), `exitCode` (number | null), `time` ({ start, end }).

2. Update `parseOpencodeLine()` in `src/main/planManager.ts` to parse `tool_use` events from opencode's JSON output and emit `tool_use` chunks.

3. Update `PlanMessage` type to include an ordered `content` array:

   ```
   content?: Array<{ kind: "text" | "thinking" | "tool_use"; content: string; data?: ToolUseData }>
   ```

   This preserves temporal ordering (e.g., thinking → tool_use → text → tool_use → text). Keep existing `text` and `thinking` fields for backward compat with stored messages.

4. Handle `tool_use` chunks in `applyChunk()` in `usePlanStore.ts` — append to the message's `content` array.

5. Handle `tool_use` chunks in `App.tsx`'s `plan:chunks` listener to ensure agent bubble exists before applying.

6. Update the log tailer (`tmuxSupervisor.ts`) to parse and forward `tool_use` lines during replay — existing log files already contain these JSON lines, they were just ignored.

**Phase 2 — UI Rendering**

7. Create `ToolUseBlock` component in its own file (`src/renderer/src/components/TaskDetail/ToolUseBlock.tsx`):
   - Collapsible block with tool-specific icon (bash: terminal, read: file, write: pencil, edit: diff, grep: search, glob: folder, task: robot, webfetch/websearch: globe)
   - Shows title and tool name when collapsed
   - Expandable to show input parameters and output
   - Visual differentiation for non-zero exit codes (red/amber accent)
   - Truncated output display with "output truncated" indicator when >5KB

8. Update `ChatMessage` in `PlanChat.tsx` to render the ordered `content` array:
   - `kind: "text"` → ReactMarkdown (current behavior)
   - `kind: "thinking"` → ThinkingBlock (current behavior)
   - `kind: "tool_use"` → ToolUseBlock (new)

9. Add styles for `ToolUseBlock` in `PlanChat.module.css` (or a co-located CSS file).

**Phase 3 — Polish**

10. Add a tool call count badge to the agent message header (e.g., "🔧 3 tool calls") when collapsed, giving immediate visibility that work is happening.

11. Consider: add `tool_use` parsing for Copilot agent as well (`parseCopilotLine()`), or document the gap as a follow-up task.

### Technical Notes

- opencode's `tool_use` event shape (from `--format json` output):

  ```json
  {
    "type": "tool_use",
    "timestamp": 1767036061199,
    "sessionID": "ses_...",
    "part": {
      "tool": "bash",
      "state": {
        "status": "completed",
        "input": { "command": "echo hello", "description": "Print hello" },
        "output": "hello\n",
        "title": "Print hello to stdout",
        "metadata": { "output": "hello\n", "exit": 0 },
        "time": { "start": 1767036061123, "end": 1767036061173 }
      }
    }
  }
  ```

- Tool types to support: `bash`, `read`, `write`, `edit`, `glob`, `grep`, `task`, `webfetch`, `websearch`

- Output truncation: cap stored output at 5KB with a `truncated: boolean` flag. Full output is not needed for the summary view.
