## Description

When Grove creates a terminal session or injects context for a task, the workspace path used for building the tmux session name, writing the context file, reading/writing the task markdown, resolving worktree paths, and loading workspace defaults — is read from the global `activeWorkspacePath` UI store state instead of from `task.workspacePath`, the authoritative workspace field every `TaskInfo` carries.

If the active workspace in the UI doesn't match the task's actual workspace (e.g., due to a race during workspace switch, or a bug in workspace-switching logic), the context file is written to workspace X's `.grove/context/` directory while the agent is running in workspace Y. The agent receives `Please read /workspace-x/.grove/context/context-{sessionName}.md` even though the task belongs to workspace Y.

### Affected locations

1. **`src/renderer/src/components/TaskDetail/TaskDetailPanel.tsx`** — `workspacePath` is derived from `activeWorkspacePath` (line 207) and then used for: `TaskTerminal` prop, `readRaw` / `writeRaw` calls, worktree path construction, and `workspaceDefaults` lookup. All of these should use the selected task's workspace.

2. **`src/renderer/src/actions/executionActions.ts`** — `startTaskExecution`, `completeTask`, and `showLaunchModalAndExecute` all snapshot `activeWorkspacePath` as `wp`. This value is used for task file read, context injection, worktree setup/teardown, and workspace defaults. All task-specific paths should use `task.workspacePath`.

### Fix direction

`task.workspacePath` is the single source of truth for which workspace a task belongs to. `activeWorkspacePath` should only be consulted for workspace-level UI actions where no task context exists. The minimal idiomatic fix in `TaskDetailPanel` is:

```ts
const workspacePath = task?.workspacePath ?? useWorkspaceStore(s => s.activeWorkspacePath);
```

This propagates the correct path through all downstream uses in the component. In `executionActions.ts`, each action function should use `task.workspacePath` directly rather than a snapshot of the global store.

## Definition of Done

- [x] `TaskDetailPanel.tsx`: `workspacePath` is derived from the selected task's `task.workspacePath` (falling back to `activeWorkspacePath` only when no task is selected). All task-specific operations in the component (`readRaw`, `writeRaw`, worktree path resolution, workspace defaults lookup, `TaskTerminal` prop) use this corrected value.
- [x] `executionActions.ts` — `startTaskExecution`: workspace path for task file read (`readRaw`), context injection (`injectExecutionContext`), and session creation (`taskterm.create`) is taken from `task.workspacePath`, not from `activeWorkspacePath`.
- [x] `executionActions.ts` — `completeTask`: worktree teardown (`git.teardownWorktreeForTask`) uses `task.workspacePath`.
- [x] `executionActions.ts` — `showLaunchModalAndExecute`: workspace defaults fetch (`fetchDefaults`) uses the task's workspace path.
- [x] If `task.workspacePath` is missing or empty where required for a task operation, the function logs an error and returns early (no silent fallback to the wrong workspace).
- [ ] Manual verification: open a task from workspace Y while workspace X is active; confirm the context file is written to `{workspaceY}/.grove/context/` and the tmux session name hash matches workspace Y's path.
- [x] Workspace-level operations that legitimately need `activeWorkspacePath` (sidebar navigation, workspace settings, new task creation) are not affected.
