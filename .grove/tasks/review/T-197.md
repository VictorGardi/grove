## Description

When a task is in backlog (plan mode) with the agent tab open, moving it to "doing" causes the terminal session to flicker and show a "start new session" button instead of automatically starting an execution session.

**Root cause — primary (`TaskDetailPanel.tsx:694`):**
`TaskTerminal` is keyed by `task.id` only, so the component does **not** remount when `sessionMode` changes (plan → exec on backlog → doing). The initialization `useEffect` (empty deps `[]`) only fires on mount, so the exec auto-start branch (`else if (sessionMode === "exec") startNewSession()`) never re-runs. The result is `sessionState === "none"` with `sessionMode === "exec"`, which renders the "start new session" button.

**Root cause — secondary (`TaskTerminal.tsx:637` and `readFreshFrontmatter`):**
The reconnect logic hardcodes `fresh.terminalExecSession` (and `fresh.terminalExecContextSent`), and `readFreshFrontmatter` only parses exec-mode frontmatter fields. When a plan-mode session reconnects, it reads the wrong field and `contextSentRef` is left stale — potentially re-injecting context that was already sent.

**Edge case:**
`startTaskExecution` calls `window.api.taskterm.create` for an exec session before the task status update lands. When the React key change triggers a remount, the component may also call `startNewSession()` simultaneously — two competing session creation calls. This needs verification and a guard if needed.

## Definition of Done

- [x] In `TaskDetailPanel.tsx:694`, the `TaskTerminal` key is changed from `task.id` to `` `${task.id}-${sessionMode}` `` so the component remounts when sessionMode changes
- [x] In `TaskTerminal.tsx`, `readFreshFrontmatter` is extended to also parse `terminalPlanSession` and `terminalPlanContextSent` from frontmatter (in addition to the existing exec fields), and line 637 uses the correct field based on the current `sessionMode`
- [x] Moving a task from backlog to doing while the agent tab is open automatically starts an execution session (no manual click required)
- [x] Moving a task from doing back to backlog while the agent tab is open shows the plan-mode "start new session" screen (plan sessions require manual start; the plan session was killed by `startTaskExecution`)
- [x] The xterm.js terminal DOM is cleanly unmounted and re-mounted on key change with no orphaned PTY listeners or xterm instances remaining from the previous mount
- [x] Verify (and guard if needed) that moving to "doing" does not trigger two simultaneous exec session creation calls (one from `startTaskExecution`, one from the component's auto-start)
