### Part 3 — OS-level sandbox for all three agents on macOS (enabled by default, opt-out)

**Scope: macOS only.** Sandboxing uses Apple Seatbelt (`sandbox-exec`) on macOS. Linux support (bubblewrap) is out of scope and left for a follow-up task. On Linux, Grove launches agents without any sandbox wrapping regardless of the `sandboxEnabled` setting.

Every interactive agent launch on macOS is wrapped in an OS-level sandbox restricting filesystem access to the task's git worktree. Sandboxing is **enabled by default** but can be disabled globally via a workspace-level setting for users who need to run Grove without it (e.g., working on repos that span multiple directories).

**Strategy per agent (macOS):**

| Agent | Mechanism | Additional |
|-------|-----------|------------|
| `claude` | Claude Code's built-in sandbox via `.claude/settings.json` with `sandbox: { enabled: true }` in worktree root — Claude Code activates Seatbelt internally | Verify `sandbox.enabled` is the correct current Claude Code `settings.json` API during implementation |
| `opencode` | `sandbox-exec -f <profile>` wrapping the `opencode` binary invocation inside the `bash -c` command string | Also set `external_directory: "deny"` in opencode config for defense-in-depth |
| `copilot` | `sandbox-exec -f <profile>` wrapping the `copilot` binary invocation inside the `bash -c` command string | Investigate whether `COPILOT_HOME` or another mechanism provides per-task session isolation; document what is used |

**Critical implementation detail — where the sandbox wrapper goes:**
The sandbox must wrap the agent binary, not the surrounding bash process. The `taskTerminal.ts` command is passed as a string to `bash -c`. The correct structure is:
```bash
bash -c "sandbox-exec -f /tmp/grove-sandbox-<sessionId>.sb opencode run ..."
```
NOT `sandbox-exec ... bash -c "opencode ..."` (which sandboxes bash and may or may not inherit to children).

**New module: `src/main/sandbox/profileGenerator.ts`**
- `getSandboxPrefix(agent, worktreePath, sessionId)` → returns the `sandbox-exec` command prefix string and a cleanup callback; returns `null` on non-macOS platforms (no-op)
- macOS SBPL profile for opencode/copilot allows: worktree (rw), agent binary + XDG config/data/cache dirs, `/opt/homebrew/`, `/usr/`, `/System/`, `/Library/`, git config (ro), `$TMPDIR`; denies all other home directory reads
- Temp `.sb` profile files are named `grove-sandbox-<sessionId>.sb`; Grove scans and removes stale profiles at startup to handle abnormal exits

**`opencodeConfig.ts` must be redesigned** for Part 3: currently it overwrites the config file and deletes it on cleanup. It must instead read-patch-write (merge Grove's settings into any existing user config) and on cleanup restore the original content rather than delete the file. This prevents destroying a user's existing opencode configuration.

**`.claude/settings.json` placement:** Determine during implementation whether the sandbox config should be written to the worktree root (risk: committed to the task branch) or to the workspace's `.claude/` directory (outside the worktree). Add `.claude/settings.json` to the workspace `.gitignore` if writing to the worktree, or verify Claude Code's config lookup order to confirm writing to workspace root works.

**macOS `sandbox-exec` deprecation:** `sandbox-exec` is marked deprecated on macOS 15 (Sequoia) but remains functional. Document this risk in code comments. If Apple removes it, the migration path is bubblewrap-for-mac via Homebrew.

**Workspace-level toggle.** A boolean `sandboxEnabled` (default: `true`) is added to `WorkspaceEntry`. The workspace settings UI exposes this as an on/off toggle with a clear description of what it does. When disabled, all sandbox wrapping and `.claude/settings.json` sandbox config generation are skipped — agents launch without restrictions. On Linux, this toggle has no effect (sandbox is never applied).

**Worktree required when sandbox is enabled on macOS.** A task with `useWorktree: false` (or no worktree yet assigned) cannot start an agent when sandbox is on. The UI shows a clear error explaining that sandboxing requires a worktree. When sandbox is disabled, the worktree restriction does not apply.
