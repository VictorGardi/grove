## Description

Two bugs in the **Workspace Defaults** section of Settings (`AgentModelForm`):

### Bug 1 ‚ÄĒ Model dropdown permanently disabled after switching agents or workspaces

**Symptom:** After selecting a workspace in settings and choosing an agent (e.g. opencode ‚Üí copilot), the model dropdown remains disabled. Navigating to another workspace and back "fixes" it temporarily.

**Root causes (two separate issues):**

1. **Effects-ordering race in `AgentModelForm.tsx`:** The component tracks `planningModelsLoading` and `planningModels` as redundant local state that is managed by two conflicting effects. When `planningAgent` changes and the model cache for the new agent already exists (e.g. `[]` from a prior fetch), both effects fire in the same React cycle. React runs effects in definition order: the "sync from cache" effect (Effect 2) fires first and clears loading, then the "reset on agent change" effect (Effect 3) fires and sets loading back to `true`. Since the cache entry hasn't changed, Effect 2 never re-fires. Loading is permanently stuck at `true`, keeping the dropdown disabled.

   **Fix:** Remove the redundant "reset on agent change" Effects 3 & 4. Derive `planningModels`, `planningModelsLoading`, `executionModels`, and `executionModelsLoading` directly from the cache entry:
   - `models = Array.isArray(cacheEntry) ? cacheEntry : []`
   - `loading = !Array.isArray(cacheEntry)` ‚ÄĒ `undefined` (not fetched) and `null` (in-flight) both mean loading; an array (even empty) means done.

2. **`initialized` flag not reset on workspace change:** Effect 6 in `AgentModelForm` calls `fetchDefaults(workspacePath).then(() => setInitialized(true))`, but never resets `initialized` to `false` first when `workspacePath` changes to a new valid path. If `initialized` was already `true`, `setInitialized(true)` is a no-op, so Effect 7 (which applies workspace defaults to local state) does not re-run with fresh data. Saved agent/model selections for the new workspace are silently ignored.

   **Fix:** Add `setInitialized(false)` before the async `fetchDefaults` call in Effect 6. Also guard against `fetchDefaults` failure: set `initialized=true` even on error (so the form doesn't silently hang in a non-initialized state).

### Bug 2 ‚ÄĒ OpenCode models never appear in the dropdown

**Symptom:** When `opencode` is selected as the agent, the model dropdown shows only "Default" ‚ÄĒ no models are listed, even when opencode is installed.

**Root causes:**

1. **Parser mismatch:** `spawnOpencodeModels` runs `opencode models` and parses stdout with the regex `/\b([a-zA-Z0-9_-]+\/[a-zA-Z0-9._-]+)\b/g` looking for `provider/model` patterns. The actual output format of `opencode models` has not been verified against this regex. If the format differs, the parser silently produces `[]`.

2. **No subprocess timeout:** `spawnOpencodeModels` has no timeout. If `opencode` is slow to start or hangs, the model fetch never resolves and loading is stuck indefinitely.

3. **Permanent empty cache:** Once the cache entry for `workspacePath:opencode` is set to `[]`, `ensureModels` returns immediately on every subsequent call. There is no way to retry without restarting the app.

   **Fix:** Verify `opencode models` output format and update the parser if needed. Add a ‚Č§15s timeout (with `proc.kill()` on timeout to avoid zombie processes). Add a "Refresh models" button that clears the cache entry for the current workspace+agent and re-triggers the fetch.
