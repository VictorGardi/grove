## Description

### Root Cause

There are three cascading bugs in `src/renderer/src/App.tsx`, all in the session restoration `useEffect` (lines 118–153):

**Bug 1 — Missing terminal session fields in initialization guards.**
The loop calls `usePlanStore.initSession` only when `task.planTmuxSession || task.planSessionId` (plan) or `task.execTmuxSession || task.execSessionId` (exec) are non-null. Tasks started via `TaskTerminal` store their tmux session name in `task.terminalPlanSession` / `task.terminalExecSession` — not in those legacy fields — so `initSession` is never called for them. This leaves `usePlanStore.sessions["plan:<taskId>"]` as `undefined`.

**Bug 2 — Polling guard skips undefined and idle sessions.**
`checkTmuxSessions` in `useWorkspaceStatus.ts` (line 146) has `if (!session || session.isRunning) continue`. With session `undefined`, the tmux liveness check is skipped entirely. Additionally, `initSession` called with a `null` existingSessionId initialises `sessionStatus: "idle"` (usePlanStore.ts line 182), and the guard at lines 148–151 also skips sessions with `sessionStatus === "idle"`. So even after fixing Bug 1, the polling would still skip the task unless `sessionStatus` is explicitly set to `"paused"` after initialisation.

**Bug 3 — `review` tasks excluded from initialisation.**
The `dashboardTasks` filter (line 125–127) only includes `doing` and `backlog`, but `useWorkspaceStatus.ts` polls `review` tasks too. Review tasks with active terminal sessions are never initialised and never show indicators.

**Bug 4 — Indicator rendering in TaskCard.tsx only checks legacy fields.**
The "waiting for input" indicator logic checks `task.planTmuxSession` / `task.execTmuxSession` but not `task.terminalPlanSession` / `task.terminalExecSession`. So even when tmux is alive, the indicator won't show for terminal-based tasks.

### Fix Required

1. `src/renderer/src/App.tsx` (lines 118–170):
   - Add `|| t.status === "review"` to the `dashboardTasks` filter (line 125).
   - Add `|| task.terminalPlanSession` to the plan mode guard (line 130).
   - Add `|| task.terminalExecSession` to the exec mode guard (line 142).
   - After calling `initSession` for a task where `existingSessionId` is `null` but `terminalPlanSession` / `terminalExecSession` is set, immediately call `usePlanStore.getState().setSessionStatus(key, "paused")` so the tmux polling guard passes.

2. `src/renderer/src/components/Board/TaskCard.tsx` (lines 81, 93, 104):
   - Add `|| task.terminalExecSession` to the exec session checks (for `isExecuteWaiting` and `isExecuteWaitingWithTmuxAlive`).
   - Add `|| task.terminalPlanSession` to the plan session check (for `isPlanWaitingWithTmuxAlive`).

### Known Limitation (out of scope)

`TaskInfo` has no `terminalPlanLastExitCode` / `terminalExecLastExitCode` fields. The "session failed" indicator for terminal sessions cannot be restored after an app restart. Acknowledged as a pre-existing limitation; not addressed here.
