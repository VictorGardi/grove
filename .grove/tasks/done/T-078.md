**BUG FOUND**: The indicator shows on ALL workspaces instead of just the workspace with active sessions. See "Bug Investigation" section below.

## Description

Add a small circular dot indicator on each sidebar workspace row (in `WorkspaceItem.tsx`) that shows the aggregate agent status across all tasks in that workspace. The indicator communicates at a glance whether agents are actively working, waiting for user input, or have encountered failures — without requiring the user to open the task board or detail panels.

### Status definitions (priority order: failed > waiting > running > idle)

- **failed** (red `--status-red`): At least one task's agent session exited with `lastExitCode !== 0`. Determined synchronously from `usePlanStore`.
- **waiting** (amber `--status-amber`): At least one task has a live tmux session, `isRunning === false`, and the session has previously executed (i.e., `isRunning` was `true` at some point, or `sessionStatus` is `'paused'`). This means the agent completed a turn and is awaiting the next user message. Determined by combining store state with a tmux liveness check.
- **running** (green `--status-green`): At least one task has `isRunning === true` in its plan session. Determined synchronously from `usePlanStore`. Optionally add a subtle CSS pulse animation for visual distinction from the terminal tab's green dot.
- **idle** (no indicator): No active sessions, or all sessions are idle with no tmux sessions alive.

### Derivation approach

Compute entirely in the renderer using existing stores:

- `useDataStore.tasks` — provides task list with `planTmuxSession` / `execTmuxSession` names
- `usePlanStore.sessions` — provides `isRunning`, `lastExitCode`, `sessionStatus` per `mode:taskId` key

The only external signal needed is tmux session liveness, checked via the existing `plan:checkTmuxSession` IPC handler. This is only polled when `isRunning === false` and the session has previously run — to distinguish "waiting for input" from "never started."

### Priority resolution

1. Within a single task: check both plan and exec sessions, take highest-priority status
2. Across the workspace: take the highest-priority status among all tasks

### UI specification

- 8px circular dot positioned to the right of the workspace name, before the existing `[N]` count badge
- Colors from existing design system: red `#e05c5c`, amber `#e8a44a`, green `#3ecf8e`
- No dot rendered for `idle` status (zero visual change from current behavior)
- Tooltip on hover:
  - failed: `N task(s) failed`
  - waiting: `N task(s) waiting for input`
  - running: `N agent(s) running`
- Running indicator: optional subtle pulse animation (CSS `@keyframes` opacity oscillation, ~2s cycle)

### Files to modify

1. `src/renderer/src/components/Sidebar/WorkspaceItem.tsx` — add indicator dot, tooltip, and derive status
2. `src/renderer/src/stores/useWorkspaceStore.ts` — or create a new hook/composable for the derived status computation with tmux polling
3. Verify `plan:checkTmuxSession` is exposed in `src/preload/index.ts` and typed in `src/preload/index.d.ts`

### Edge cases

- Multiple tasks with different statuses → show highest priority, count reflects that status group
- Task moved from doing to done → its session is cleared, indicator updates immediately
- App restart with orphaned tmux sessions → reconnect logic already exists; indicator reflects reconnected state
- Workspace with no doing tasks → no indicator (idle)
- IPC checkTmuxSession failure → degrade to `idle` with console warning, do not block rendering
- Concurrent plan + exec sessions on same task → highest-priority session wins within task
- `waiting` false positive guard: only show waiting if the session has previously run (check `sessionStatus !== 'idle'` initially, or track that `isRunning` was ever `true`)

## Bug Investigation (2026-04-07)

### Problem

The green running indicator is showing on ALL workspaces even though only ONE workspace has active agent sessions.

### Root Cause

The `useWorkspaceStatus` hook in `src/renderer/src/hooks/useWorkspaceStatus.ts` receives a `workspacePath` parameter but does NOT filter tasks by that path. At line 68 it does:

```typescript
const tasks = useDataStore((s) => s.tasks);
```

This returns ALL tasks from the data store (which contains tasks only for the active workspace). Then at line 174:

```typescript
const doingTasks = tasks.filter((t) => t.status === "doing");
```

This filters to doing tasks but STILL uses all tasks from the store. Since `useDataStore` is shared global state containing only the active workspace's tasks, every workspace row gets the same doing tasks.

Additionally, `usePlanStore.sessions` (line 69) contains sessions for ALL tasks across ALL workspaces (since it's keyed by `mode:taskId` and not cleared on workspace switch), so it's not filtered by workspace either.

### Why It Shows on All Workspaces

1. When workspace A is active, `useDataStore.tasks` contains workspace A's tasks
2. All workspace rows (A, B, C) call `useWorkspaceStatus(theirPath)`
3. All of them get `useDataStore.tasks` = workspace A's tasks (same reference)
4. All of them get `usePlanStore.sessions` = global sessions (not filtered by workspace)
5. Since workspace A has running sessions, ALL workspace rows derive `status: "running"`

### Fix Required

The `useWorkspaceStatus` hook MUST filter tasks by `workspacePath`. But `TaskInfo` does NOT have a `workspacePath` field - it only has `filePath` which is relative to the workspace.

**Option 1**: Add `workspacePath: string` to `TaskInfo` interface in `src/shared/types.ts`. Populate it in the main process when scanning tasks.

**Option 2** (simpler): Derive workspace path from `task.filePath` - since `filePath` is relative to workspace root (e.g., `.tasks/doing/T-001.md`), we can infer the workspace by checking if `filePath` starts with known patterns. But this doesn't work because the same file path could exist in multiple workspaces.

**Option 3** (current architecture fix): Since `useDataStore.tasks` only contains tasks for the ACTIVE workspace, the indicator should ONLY be computed for the ACTIVE workspace. For inactive workspaces, we need either:

- A separate data source that tracks tasks per workspace (requires main process changes)
- OR only show the indicator on the active workspace row

**Recommended Fix**: Add `workspacePath` field to `TaskInfo` so each task knows which workspace it belongs to. Then filter in `useWorkspaceStatus`:

```typescript
const doingTasks = tasks.filter(
  (t) => t.status === "doing" && t.workspacePath === workspacePath,
);
```

Similarly for sessions - filter by workspace via task IDs that belong to that workspace.
