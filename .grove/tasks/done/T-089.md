## Description

After the app is reloaded/restarted, no indicators (agent running, waiting for input, session failed) are showing on any tasks in a workspace, even though they were active before the reload. The user needs to click on a task and close it to see the indicator. This needs to be fixed.

### Root Cause

The `usePlanStore` is an in-memory Zustand store that starts empty when the app restarts. When a task was running before reload, its session info (`isRunning`, `sessionStatus`, `lastExitCode`, etc.) is lost from the store. Session initialization happens in `PlanChat.tsx` — but `PlanChat` only mounts when a task is opened in the detail panel. Since the board/TaskCard components don't have access to the session state until `PlanChat` runs, no indicators are shown until the user explicitly opens and closes a task.

### Proposed Solution: Store Exit Code in Task Frontmatter (Option 3)

Instead of replaying logs to get exit code, persist `lastExitCode` directly to task frontmatter. This allows full indicator support on reload without needing to open tasks.

#### Implementation Steps

1. **Add new frontmatter fields** (in `src/shared/types.ts` and `src/main/tasks.ts`):
   - Add `planLastExitCode?: number | null` to `TaskInfo` interface
   - Add `execLastExitCode?: number | null` to `TaskInfo` interface
   - Add same fields to `TaskFrontmatter` interface
   - Update `buildFrontmatter()` to include these fields when writing tasks
   - Update `parseTaskFile()` to read these fields from frontmatter

2. **Write exit code when agent completes** (in `src/main/planManager.ts` or `src/main/tmuxSupervisor.ts`):
   - When `__grove_exit` sentinel is parsed (around line 376 in tmuxSupervisor.ts), extract exit code
   - Add a new IPC handler `task:updateExitCode(taskId, mode, exitCode)` to persist the exit code
   - Or reuse existing task update pattern to write the field to frontmatter

3. **Initialize sessions at board level** (in `src/renderer/src/App.tsx` after tasks load):
   - For each task in "doing" or "backlog" with `planTmuxSession` or `execTmuxSession`:
     - Call `usePlanStore.getState().initSession(sessionKey, agent, model, sessionId)`
     - Set `lastExitCode` from `task.planLastExitCode` / `task.execLastExitCode` (new fields)
     - Set `isRunning: false` (tmuxCheck will detect if it's actually running)
   - Only process "doing" and "backlog" tasks for performance

#### Expected Behavior After Fix

| Scenario                           | Indicator Shown                        |
| ---------------------------------- | -------------------------------------- |
| Agent was running → still running  | "agent running" (via tmuxCheck)        |
| Agent finished cleanly (exit 0)    | "Waiting for input" (from frontmatter) |
| Agent failed (exit non-zero)       | "session failed" (from frontmatter)    |
| Agent crashed (no exit code saved) | No indicator (graceful degradation)    |

#### Edge Cases

- **Agent killed externally**: No `grove_exit` sentinel written → no exit code in frontmatter → no indicator (acceptable)
- **Crash before done chunk**: Exit code not persisted → graceful degradation (no indicator)
- **Task moved to done while running**: Exit code written to original location (acceptable)
- **Backward compatibility**: Old tasks without new fields default to `null` → no indicator

## Additional Fix Applied

The exit code persistence wasn't enough - `TaskCard` also required `messages.length > 0` to show indicators. After reload, messages is empty `[]`.

**Final fix**: For restored sessions (messages.length === 0), if task has stored tmuxSession, default to showing "Waiting for input" indicator. This handles the case where exit code wasn't persisted (e.g., older tasks or tasks where agent was killed externally). The indicator proves the session was active before reload.

**Note**: "session failed" only shows when we have explicit non-zero exit code (from store or task frontmatter).
