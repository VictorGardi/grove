## Description

Allow users to customize specific parts of the plan/execute/review prompts **per workspace**. This feature extracts parametrized parts from the existing hardcoded prompts in `planPrompts.ts` and exposes them as editable fields in Settings.

### Prompt Configuration Keys

Each key has a default value. If a user leaves a field empty or resets it, the default is used.

| Config Key                  | Description                              | Default Value                | Example                       |
| --------------------------- | ---------------------------------------- | ---------------------------- | ----------------------------- |
| `planPersona`               | Role name for the planning agent         | `"planning assistant"`       | `"senior data engineer"`      |
| `planReviewPersona`         | Role name for the plan reviewer          | `"senior software engineer"` | `"principal engineer"`        |
| `executePersona`            | Role name for the execution agent        | `"execution agent"`          | `"implementation specialist"` |
| `executeReviewPersona`      | Role name for the execution reviewer     | `"senior software engineer"` | `"principal engineer"`        |
| `executeReviewInstructions` | Custom instructions for the review phase | See default text below       | `"Focus on security..."`      |

**Default `executeReviewInstructions`** (extracted from current prompt lines 69-83):

```
IMPORTANT: After you have checked off all DoD items in the task file, BEFORE stopping your execution (exiting the session), you MUST spawn a {executeReviewPersona} subagent to review your code changes. Use the subagent to:

1. Review the actual code changes (via `git diff` or equivalent)
2. Verify each DoD item was genuinely implemented (not just checked off)
3. Check for edge cases and code quality issues
4. Address any issues found before stopping

You may run up to 2 review cycles to avoid infinite loops. When spawning the reviewer, follow these rules:

- Spawn a dedicated reviewer subagent ({executeReviewPersona}). Record and increment a numeric `reviewCycleCount` in your session state each time you auto-spawn the reviewer.
- The reviewer MUST inspect the actual diffs (for example by running `git diff` in the workspace) and verify each DoD item was implemented. The reviewer should list which DoD items are satisfied and which are not, referencing file paths and diff hunks.
- The reviewer MUST emit an explicit session-end event that clearly indicates PASS or FAIL for the review (e.g. `SESSION-END: PASS` or `SESSION-END: FAIL`). The execution agent should detect this and act accordingly.
- If the reviewer signals FAIL, address the raised issues and you may re-run the reviewer, but stop auto-spawning after `reviewCycleCount == 2`.

Only stop (exit) your session after the reviewer emits a PASS or after you have exhausted the allowed review cycles and have no further automated actions to take.
```

### Implementation Approach

1. **Data Model** — Add 5 optional fields to `WorkspaceDefaults` in `useWorkspaceStore.ts` and persist via existing `workspace:setDefaults` IPC.

2. **Prompt Builder Refactoring** — Refactor `planPrompts.ts`:
   - Define default constants in one place
   - `buildFirstPlanMessage` and `buildFirstExecutionMessage` accept an optional `prompts` parameter
   - Template variables are replaced: `{planPersona}` → `config.planPersona ?? DEFAULT_PLAN_PERSONA`
   - Persona names are injected wherever "planning assistant", "senior software engineer", "execution agent" appear

3. **UI** — Add a new `WorkspacePromptsForm` component in Settings:
   - 5 text input fields, each with a "Reset to Default" button
   - Character limit: 500 characters per field
   - Security warning: "Prompt content is sent to external agents. Be cautious of instructions that could override system behavior."

4. **Integration** — In `PlanChat.tsx`, fetch workspace prompts from store and pass to prompt builders.

### Edge Cases

- **Empty field** — Treat as empty string, fall back to default value
- **Config migration** — Gracefully handle workspaces with missing prompt fields (use defaults)
- **Workspace removed** — Prompt config persists until user removes workspace from config
- **Corrupted config** — Existing config validation already handles this (parse error → defaults)

### Security Considerations

- Add a warning in the UI about prompt injection risks
- No character-level sanitization (trust the user)
