## Description

Epic: integrate opencode's SDK as a first-class execution backend for Grove tasks, replacing the tmux+TUI path for opencode agents. One shared `opencode serve` process handles all worktrees. Sessions persist across Grove restarts. Grove streams typed SDK events into a rich React UI instead of raw xterm.js terminal output.

Split into four sub-tasks (each independently executable in order):

- **T-200** ŌĆö opencode server lifecycle & preload bridge
- **T-201** ŌĆö opencode session management IPC
- **T-202** ŌĆö SSE event bridge (main process)
- **T-203** ŌĆö Rich event UI (`TaskEventStream` + execution flow)
- **T-204** ŌĆö Startup reconciliation, agent selector & worktree teardown cleanup

## Key architecture decisions (informed by review)

- **One server, not one per worktree**: `opencode serve` accepts `directory` per API call; a single process serves all worktrees. T-200 manages this singleton.
- **SDK client in `src/main/`**: not in `src/shared/` ŌĆö the SDK uses Node-only APIs and must not be bundled into the renderer.
- **Prompt built in renderer**: `buildFirstPlanMessage`/`buildFirstExecutionMessage` live in `src/renderer/src/utils/planPrompts.ts` and cannot be imported by main. The renderer builds the prompt string and passes it to `opencodeSession:prompt` as a pre-built string.
- **`message.done` does not exist**: `session.status { type: "idle" }` signals turn completion. Grove-level task completion remains an explicit user action.
- **Message history replay via `client.session.messages()`**: `opencodeSession:get` only returns session state; message history for remount replay requires a separate `opencodeSession:messages` IPC call.
- **Preload bridge required**: all new IPC calls must be added to `src/preload/index.ts`.
- **Event batching**: events relayed from main to renderer are batched over 16 ms to reduce structured-clone overhead for high-frequency streaming.
- **`permission.asked` must surface to UI**: silently dropping it stalls the agent.
