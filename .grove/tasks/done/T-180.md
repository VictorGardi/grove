## Description

### Bug Analysis

After thorough code review of `src/renderer/src/stores/useTaskSwitcherStore.ts`, I identified **three critical bugs** and one **design issue**:

---

### Bug 1: `toggleLastTask` does not correctly track the "previous" task

**Location:** Lines 100-125, specifically lines 111-117

**Current buggy code:**

```typescript
set({
  lastTaskId: state.lastTaskId, // BUG: keeps current task unchanged!
  recentTaskIds: [
    state.lastTaskId!,
    ...state.recentTaskIds.filter((id) => id !== state.lastTaskId),
  ].slice(0, MAX_RECENT),
});
```

**Problem:**

- `lastTaskId` is supposed to track the task we're switching AWAY from (so Cmd+E can toggle back to it)
- But the code sets `lastTaskId: state.lastTaskId` which **keeps the current value unchanged**
- This means `lastTaskId` never updates properly during toggle operations

**Impact:** Cmd+E may go to the wrong task or behave inconsistently.

**Fix:**

```typescript
toggleLastTask: (workspaces, allTasksMap) => {
  const state = get();
  const prevTaskId = state.recentTaskIds.find(
    (id) => id !== state.lastTaskId,
  );
  if (!prevTaskId) return null;

  // Verify task still exists
  for (const ws of workspaces) {
    const tasks = allTasksMap.get(ws.path) || [];
    const found = tasks.find((t) => t.id === prevTaskId);
    if (found) {
      // FIX: Capture lastTaskId BEFORE updating, then set it to prevTaskId
      // This correctly tracks which task we're switching FROM
      const prevLastTaskId = state.lastTaskId;
      set({
        lastTaskId: prevTaskId,
        recentTaskIds: [
          prevLastTaskId !== null ? prevLastTaskId : prevTaskId,
          ...state.recentTaskIds.filter((id) => id !== prevLastTaskId && id !== prevTaskId),
        ].slice(0, MAX_RECENT),
      });
      return prevTaskId;
    }
  }
  // ... rest unchanged
},
```

### Bug 2: `recordTaskView` overwrites `lastTaskId` too eagerly

**Location:** Line 94

**Current code:**

```typescript
lastTaskId: state.lastTaskId === taskId ? state.lastTaskId : taskId,
```

**Problem:**

- This sets `lastTaskId = taskId` whenever viewing a NEW task
- But if `toggleLastTask` already set `lastTaskId = prevTaskId`, this overwrites it with the current task
- The result is that `lastTaskId` always equals the most recently viewed task, not the task we're switching FROM

**Fix:** Simply set `lastTaskId = taskId` always (remove the conditional):

```typescript
recordTaskView: (taskId) => {
  set((state) => {
    const newRecent = [
      taskId,
      ...state.recentTaskIds.filter((id) => id !== taskId),
    ].slice(0, MAX_RECENT);
    return {
      recentTaskIds: newRecent,
      lastTaskId: taskId,  // FIX: Always update to current task
      taskLastViewedAt: { ...state.taskLastViewedAt, [taskId]: Date.now() },
    };
  });
},
```

### Bug 3: Filter logic has unreliable timestamp comparison

**Location:** Lines 220-226

**Current code:**

```typescript
filtered = filtered.filter(
  (t) =>
    t.task.status !== "done" &&
    (t.isRunning || (state.taskLastViewedAt[t.task.id] || 0) > 0),
);
```

Wait, looking more closely... actually this code is OK for the basic case. Let me re-examine the original filter:

**Original code (line 224):**

```typescript
(state.taskLastViewedAt[t.task.id] || 0) > 0;
```

**Problem:**

- This checks if `taskLastViewedAt` is greater than 0
- But this relies on `taskLastViewedAt` being properly set and persisted
- Without persistence, this resets on app restart

More importantly, there's an edge case:

- If a task was marked done BEFORE the current session
- And then viewed in a previous session (but not this one)
- On app restart, `taskLastViewedAt` is 0, so the done task won't show
- But if it was viewed in this session before being marked done, `taskLastViewedAt` > 0

This is actually **expected behavior** (show recently viewed done tasks), but the logic could be cleaner:

**Simplified fix:**

```typescript
filtered = filtered.filter(
  (t) =>
    t.task.status !== "done" ||
    t.isRunning ||
    (state.taskLastViewedAt[t.task.id] || 0) > 0,
);
```

### Design Issue: No persistence for recent task history

**Location:** The entire store (lines 49-58)

**Problem:**

- `recentTaskIds`, `lastTaskId`, and `taskLastViewedAt` are not persisted
- On app restart, Cmd+E loses all history
- This is a significant UX issue for power users

**Fix:** Add localStorage persistence following the pattern in `useNavStore`:

```typescript
const STORAGE_KEY = "grove:taskSwitcher";

interface PersistedState {
  recentTaskIds: string[];
  lastTaskId: string | null;
  taskLastViewedAt: Record<string, number>;
}

function getPersistedState(): PersistedState {
  try {
    const stored = localStorage.getItem(STORAGE_KEY);
    if (stored) {
      return JSON.parse(stored);
    }
  } catch {
    // Ignore localStorage errors
  }
  return {
    recentTaskIds: [],
    lastTaskId: null,
    taskLastViewedAt: {},
  };
}

function persistState(state: PersistedState): void {
  try {
    localStorage.setItem(STORAGE_KEY, JSON.stringify(state));
  } catch {
    // Ignore localStorage errors
  }
}

export const useTaskSwitcherStore = create<TaskSwitcherState>()((set, get) => {
  const persisted = getPersistedState();

  return {
    isOpen: false,
    searchQuery: "",
    recentTaskIds: persisted.recentTaskIds,
    lastTaskId: persisted.lastTaskId,
    taskLastViewedAt: persisted.taskLastViewedAt,
    selectedIndex: 0,
    includeDoneTasks: false,
    createWorkspaceIndex: 0,
    // ... rest of store
  };
});
```

Then update `recordTaskView` and `toggleLastTask` to persist:

```typescript
recordTaskView: (taskId) => {
  set((state) => {
    const newRecent = [
      taskId,
      ...state.recentTaskIds.filter((id) => id !== taskId),
    ].slice(0, MAX_RECENT);
    const newState = {
      recentTaskIds: newRecent,
      lastTaskId: taskId,
      taskLastViewedAt: { ...state.taskLastViewedAt, [taskId]: Date.now() },
    };
    persistState({
      recentTaskIds: newState.recentTaskIds,
      lastTaskId: newState.lastTaskId,
      taskLastViewedAt: newState.taskLastViewedAt,
    });
    return newState;
  });
},
```

## Context for agent

The core issue is that `lastTaskId` should track the task we're switching FROM, not the task we're switching TO. This requires coordination between `toggleLastTask` (which captures the current task before switching) and `recordTaskView` (which is called after switching completes).

The fix ensures:

1. `toggleLastTask` captures `prevLastTaskId = state.lastTaskId` before any update
2. `toggleLastTask` sets `lastTaskId = prevTaskId` (the task we're going TO)
3. `recordTaskView` sets `lastTaskId = taskId` (the current task after view)
4. On next `toggleLastTask`, the check `id !== state.lastTaskId` correctly finds the previous task
