### Part 3 — OS-level sandbox for all three agents (always on, no toggle)

Every interactive agent launch is wrapped in an OS-level sandbox restricting filesystem access to the task's git worktree. Sandboxing is mandatory — there is no configuration to disable it.

**Strategy per agent:**

| Agent | macOS | Linux | Additional |
|-------|-------|-------|------------|
| `claude` | Claude Code's built-in sandbox via `.claude/settings.json` with `sandbox: { enabled: true }` in worktree root — Claude Code activates Seatbelt internally | Same — bubblewrap activated internally | Verify `sandbox.enabled` is the correct current Claude Code `settings.json` API during implementation |
| `opencode` | `sandbox-exec -f <profile>` wrapping the `opencode` binary invocation inside the `bash -c` command string | `bwrap` args wrapping the `opencode` invocation | Also set `external_directory: "deny"` in opencode config for defense-in-depth |
| `copilot` | `sandbox-exec -f <profile>` wrapping the `copilot` binary invocation | `bwrap` args wrapping the `copilot` invocation | Investigate whether `COPILOT_HOME` or another mechanism provides per-task session isolation; document what is used |

**Critical implementation detail — where the sandbox wrapper goes:**
The sandbox must wrap the agent binary, not the surrounding bash process. The `taskTerminal.ts` command is passed as a string to `bash -c`. The correct structure is:
```bash
# macOS — sandbox wraps the agent:
bash -c "sandbox-exec -f /tmp/grove-sandbox-<sessionId>.sb opencode run ..."

# Linux — bwrap wraps the agent:
bash -c "bwrap --bind /worktree /worktree ... opencode run ..."
```
NOT `sandbox-exec ... bash -c "opencode ..."` (which sandboxes bash and may or may not inherit to children).

**New module: `src/main/sandbox/profileGenerator.ts`**
- `getSandboxPrefix(agent, worktreePath, sessionId)` → returns the command prefix string and a cleanup callback
- macOS SBPL profile for opencode/copilot allows: worktree (rw), agent binary + XDG config/data/cache dirs, `/opt/homebrew/`, `/usr/`, `/System/`, `/Library/`, git config (ro), `$TMPDIR`; denies all other home directory reads
- Linux bwrap spec: bind mounts for worktree, agent dirs, system paths (`/usr`, `/lib`, `/lib64`, `/etc/resolv.conf`, `/etc/ssl`); `--tmpfs /tmp`, `--dev /dev`, `--proc /proc`, `--unshare-pid`, `--die-with-parent`
- Temp `.sb` profile files are named `grove-sandbox-<sessionId>.sb`; Grove scans and removes stale profiles at startup to handle abnormal exits

**`opencodeConfig.ts` must be redesigned** for Part 3: currently it overwrites the config file and deletes it on cleanup. It must instead read-patch-write (merge Grove's settings into any existing user config) and on cleanup restore the original content rather than delete the file. This prevents destroying a user's existing opencode configuration.

**`.claude/settings.json` placement:** Determine during implementation whether the sandbox config should be written to the worktree root (risk: committed to the task branch) or to the workspace's `.claude/` directory (outside the worktree). Add `.claude/settings.json` to the workspace `.gitignore` if writing to the worktree, or verify Claude Code's config lookup order to confirm writing to workspace root works.

**macOS `sandbox-exec` deprecation:** `sandbox-exec` is marked deprecated on macOS 15 (Sequoia) but remains functional. Document this risk in code comments. If Apple removes it, the migration path is bubblewrap-for-mac via Homebrew or `@anthropic-ai/sandbox-runtime` (if it becomes a real published package).

**Worktree required.** A task with `useWorktree: false` (or no worktree yet assigned) cannot start an agent. The UI shows a clear error explaining that sandboxing requires a worktree.

**Linux fail-closed.** If `bwrap` is not installed or `CLONE_NEWUSER` is rejected at runtime (hardened kernels), agent launch fails with a user-visible error — not a crash, not a silent unsandboxed fallback.
